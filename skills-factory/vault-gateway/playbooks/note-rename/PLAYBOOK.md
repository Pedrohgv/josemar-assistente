# Note Rename Playbook

## Status

- State: active
- Mode: Hermes vault-gateway flow

## Route Mapping

- Route: `note.rename`
- Payload: `path` (required), `new_title` (required), `rewrite_wikilinks` (optional, default true)

## Purpose

- Rename a note in place when its filename is wrong (e.g. caused by a `note.capture` call where the LLM forgot to pass an explicit `title`).
- Optionally rewrite `[[old-stem]]` and `[[old-stem|alias]]` wikilinks across the vault so existing links keep working.
- Provide a gateway-level alternative to touching `/opt/data/obsidian/` directly.

## When to Use

- The note's filename is broken (truncated, missing diacritics, autogenerated timestamp, etc.) and the title needs to be human-readable.
- The note was filed in the right folder but with a meaningless name.
- A bulk-rename of a topic is needed and downstream wikilinks should follow.

## When Not to Use

- For moving a note to a different folder, prefer `note.file` (it has a different primary intent and lighter maintenance footprint).
- For content edits, use `note.update` with the appropriate mode.

## MBIFC Method Port

- Treat the new title as the source of truth for the filename. The gateway slugifies the title to derive the new file stem.
- Always prefer to rewrite wikilinks unless the user is doing a one-off rename of an internal scratch note with no inbound links.
- If the rename is part of a broader reorganization, do it in a single turn before the next move/relink step so structural context stays consistent.

## Runtime Reality (Current Implementation)

Current deterministic handler behavior:
- Resolves the source note from `path` (must be a relative markdown path inside vault root).
- Slugifies `new_title` using the same rules as `note.capture` (so the new filename is predictable).
- Refuses to no-op: if the slugified new title equals the current stem, returns a validation error.
- Generates a unique filename in the same folder if the target name is already taken.
- When `rewrite_wikilinks` is true (default), scans every other markdown file in the vault and rewrites:
  - `[[old-stem]]` → `[[new-stem]]`
  - `[[old-stem|alias]]` → `[[new-stem|alias]]` (preserves the alias text)
  - Leaves all other wikilinks untouched.
- Skips the source file itself when rewriting (its body still references the old stem in markdown but the file is being moved; the new file body will keep the same text and the gateway does not rewrite self-references in the body — body content is the user's responsibility to update).
- Skips the target file (post-move location) to avoid double-rewriting.
- After the move, refreshes `Meta/vault-structure.md` and the affected folder `_index.md` managed blocks.
- Logs the operation in `Meta/vault-gateway-log.md` with from/to paths and rewrite count.

Current handler does not:
- Update frontmatter `title` field of the moved note (the filename and H1 are the contract; if a `title` frontmatter field exists it is left untouched).
- Rewrite markdown links written as plain `[text](old-stem.md)` (only `[[...]]` wikilinks).
- Rewrite references inside `Meta/vault-gateway-log.md` (operational log) or `Meta/vault-structure.md` (snapshot, not reference).
- Detect and update bidirectional section appends or MOC entries that point to the old stem indirectly.

Those richer behaviors should be orchestrated as additional turns (`note.update` to refresh frontmatter, `note.read` to verify, etc.).

## Output Expectations

Return:
- `from` (relative path of the source)
- `to` (relative path of the destination)
- `old_stem`, `new_stem`, `new_title`
- `rewritten_notes` (list of `{path, replacements}` for transparency)
- `rewritten_count` (total wikilink replacements applied)
- `maintenance_updates` (context-block refresh entries)

## Safety

- Source path must resolve inside the vault root; absolute paths and `..` traversal are rejected by the gateway.
- Source must be an existing markdown note.
- Slugified new title must be non-empty (a title that becomes empty after slugification is rejected).
- No-op renames are rejected.
- The vault-gateway log entry is the audit trail — never rename a note by hand in `/opt/data/obsidian/`, always use this route.
